希腊机房性能实测#

完成基础信息确认后，接下来通过一系列标准化测试工具，检验这台 VPS 的真实性能表现。

CPU 性能测试#

使用 Sysbench 进行 CPU 运算能力测试，默认质数上限 10000:

root@r1089906:~# sysbench cpu --threads=1 run
CPU speed:
    events per second:   857.85
Latency (ms):
         min:                                    0.94
         avg:                                    1.16
         max:                                    9.09
         95th percentile:                        1.61

单核每秒完成 857.85 次事件，这个成绩对于 E5-2697A v4 处理器来说算是正常发挥。将难度提升到质数上限 20000 时，每秒事件数降至 282.17，性能衰减符合预期。考虑到这是 2016 年推出的老架构 CPU，不过应付日常轻量应用还是够用的。

内存性能测试#

root@r1089906:~# sysbench memory run
Total operations: 38132243 (3811903.55 per second)
37238.52 MiB transferred (3722.56 MiB/sec)
Latency (ms):

内存写入速度达到 3722.56 MiB/s，这个数据相当不错。延迟几乎为 0，说明内存响应速度很快。不过需要注意，这只是理论测试数据，实际应用中受限于单核 CPU 性能，很难完全发挥出内存带宽优势。

磁盘 IO 性能测试#

先看 4K 随机读写性能，这个指标最能反映 SSD 的真实能力:

root@r1089906:~# fio --name=randrw --rw=randrw --bs=4k
  read: IOPS=5452， BW=21.3MiB/s
    clat (usec): avg=151.46
  write: IOPS=5446， BW=21.3MiB/s
    clat (usec): avg=17.59

4K 随机读取 IOPS 为 5452，写入 IOPS 为 5446，读写速度均为 21.3 MiB/s。这个成绩说实话有点尴尬——虽然官方宣称使用 NVMe 固态硬盘，但从测试数据来看，更像是 SATA SSD 的表现。

再看顺序读写性能:

root@r1089906:~# fio --name=seqrw --rw=readwrite --bs=1M
  read: IOPS=752， BW=753MiB/s
  write: IOPS=801， BW=801MiB/s

顺序读取速度 753 MiB/s，写入速度 801 MiB/s，这个数据倒是能达到中端 NVMe 的水平。结合两项测试来看，这块硬盘的实际性能更接近高性能 SATA SSD 或者入门级 NVMe，并非主流认知中的高端 NVMe 硬盘。不过这个 IO 性能已经够用了。

下载速度 393.43 Mbps，上传速度 414.03 Mbps，测试目标服务器位于希腊拉里萨。考虑到官方标注的是 300Mbps Fair-share 带宽，实际测速能跑到接近 400Mbps 属于超预期表现。空闲延迟 7.59ms，丢包率 0%，说明机房内部网络质量不错。

不过需要说明的是，这只是本地希腊服务器的测速结果。对于国内用户来说，更关心的是连接中国大陆的实际速度，这个会在后续网络测试部分详细展开。Fair-share 模式意味着高峰期可能会有速度波动，但从测试时段来看，带宽充足没有明显限速。

总体来说，这台 VPS 的性能中规中矩:CPU 算力偏弱，内存表现优秀，磁盘 IO 够用但谈不上顶级，网络带宽给的很足。

连接中国网络表现#

对于国内用户来说，欧洲机房最关键的指标就是连接中国大陆的网络质量。下面从延迟、丢包、路由、实际带宽等多个角度。

回国延迟测试#

先用最简单的 ping 命令测试连接百度的延迟:

64 bytes from 103.235.46.115: icmp_seq=2 ttl=54 time=272 ms
64 bytes from 103.235.46.115: icmp_seq=3 ttl=54 time=272 ms

rtt min/avg/max/mdev = 271.910/271.936/271.961/0.020 ms

平均延迟 272ms，数据非常稳定，抖动几乎为零。这个延迟在欧洲机房中属于正常水平，不算特别优秀但也不差。

中国三网延迟分布#

通过 itDog 平台测试国内各地三大运营商的延迟情况:

从测试结果来看:

  • 电信线路表现最好，平均延迟 234ms，河北石家庄最快 198ms，新疆乌鲁木齐最慢 292ms
  • 移动线路平均延迟 283ms，上海节点 182ms，西藏拉萨 374ms
  • 联通线路最慢，平均延迟 310ms，天津 276ms，海南海口 346ms

整体来说，电信用户体验会好一些，联通用户可能会觉得延迟有点高。地理位置对延迟影响明显，西部地区普遍比东部沿海慢 50-100ms。对于网页浏览、API 调用这类应用，272ms 的延迟基本可以接受。

网络丢包率测试#

丢包率测试结果为 5%，这个数据有点高了。测试时间是中午，属于网络使用高峰期，丢包情况可能会更严重一些。5% 的丢包率对于 TCP 连接影响不算太大，因为会自动重传，但对于实时性要求高的应用(比如视频通话、在线游戏)就会有明显卡顿感。

这个丢包率和机房本身关系不大，更多是国际出口拥堵导致的。希腊到中国的海底光缆线路相对冷门，不像主流欧洲节点那样有多条备用线路，高峰期容易出现拥堵。如果是凌晨或者深夜时段测试，丢包率应该会低很多。

回国路由追踪#

从中国上海出发的数据包，先经过荷兰阿姆斯特丹，然后转到保加利亚索非亚，最终到达希腊塞萨洛尼基。这个路由并不算理想——绕了一大圈，先北上到荷兰，再折返到巴尔干半岛。如果能直连地中海海底光缆，延迟应该能降低 30-50ms。

这条路由说明 JustHost 的上游服务商在欧洲内部选择了成本优先的线路，而非延迟最优的路径。不过考虑到价格因素，这也可以理解。毕竟 6 美元一个月的 VPS，不能指望用上 CN2 GIA 这种优质线路。

平均下载速度 36.6 Mbps，峰值一度达到 61.9 Mbps。考虑到 272ms 的高延迟和 5% 的丢包率，能跑到这个速度已经不错了。特别需要注意的是重传次数高达 1858 次，说明丢包问题确实存在，TCP 协议在不停地重传丢失的数据包。

对于日常使用来说，36 Mbps 的速度下载个几百 MB 的文件还算流畅，但如果要传输 GB 级别的大文件，就会比较煎熬。速度波动也比较明显，从 30 Mbps 到 60 Mbps 跳来跳去，说明网络质量不够稳定。

补充测试项目#

IP 质量检测#

IP 归属显示为美国 ASNET(AS26383)，但实际地理位置在希腊塞萨洛尼基。所有数据库都识别为机房 IP，风险评分普遍较低，仅 ipapi 标记为中等风险(4.36%)。IP 没有被标记为代理、VPN 或 Tor 节点，滥用记录干净。

邮局连通性方面，25 端口可用，但部分邮件服务商(如 QQ、GMX、Sohu、Sina)连接受限。黑名单检测显示 439 个数据库中有 46 个标记，但没有进入实际黑名单，整体质量还算不错。

流媒体解锁测试#

流媒体解锁情况比较尴尬:

  • Netflix 可以解锁希腊区内容，但仅限自制剧
  • YouTube Premium 识别为美国区，可正常使用
  • Disney+ 直接被 IP 封禁
  • Amazon Prime Video 显示服务不可用
  • Spotify 注册被屏蔽
  • TVBAnywhere+ 可以观看

AI 服务方面，ChatGPT、Google Gemini、Claude 都能正常访问。Steam 货币显示为欧元区，Apple 服务识别为俄罗斯区。总体来说，这个 IP 的流媒体解锁能力较弱，不适合用来看剧，更适合做开发测试或建站用途。

测评总结#

经过一系列测试，JustHost 希腊塞萨洛尼基机房的整体表现基本符合价格定位。

从硬件角度看，虽然 CPU 采用了较老的 E5-2697A v4 架构，单核性能偏弱，但内存响应速度快，磁盘 IO 表现稳定。官方宣传的 NVMe 硬盘在实际测试中更接近高性能 SATA SSD 水平，顺序读写不错但随机 IOPS 一般。
