JustHost 在圣保罗部署的数据中心采用 NVMe 固态硬盘和高质量网络线路，理论上能为用户提供稳定的服务体验。

基础系统信息#

操作系统版本#

测试服务器运行的是 Ubuntu 22.04.5 LTS 长期支持版本，内核版本为 5.15.0-122-generic。这个系统版本相对稳定，软件包更新及时，适合长期运行的生产环境。从系统信息来看，JustHost 提供的是标准的 Linux 环境，没有过多定制化配置。

root@r1089906:~# lsb_release -ds
Ubuntu 22.04.5 LTS
root@r1089906:~# uname -a
Linux r1089906 5.15.0-122-generic #132-Ubuntu SMP Thu Aug 29 13:45:52 UTC 2024 x86_64 x86_64 x86_64 GNU/Linux

系统架构为 x86_64，这是目前服务器领域的主流架构，软件兼容性较好。

CPU 配置#

处理器采用的是 Intel Xeon E5-2699 v4，主频 2.20GHz。这是英特尔 Broadwell 架构的服务器级处理器，虽然不算最新一代，但稳定性和性能表现都比较成熟。单核心配置符合 Sirius 套餐的标准规格。

root@r1089906:~# lscpu | egrep 'Model name|Socket|Core|Thread|MHz'
Model name:                           Intel(R) Xeon(R) CPU E5-2699 v4 @ 2.20GHz
Thread(s) per core:                   1
Core(s) per socket:                   1
Socket(s):                            1

从配置来看，分配的是 1 个物理核心，没有开启超线程。对于轻量级应用和小型网站来说，这个 CPU 配置基本够用，但运行计算密集型任务可能会有些吃力。

内存状况#

实际可用内存为 957 MB，接近标称的 1024 MB（系统会占用一部分）。值得注意的是，JustHost 默认没有配置 Swap 交换分区，这意味着内存用尽后系统可能会出现进程被杀的情况。

root@r1089906:~# free -h
               total        used        free      shared  buff/cache   available
Mem:           957Mi       182Mi       174Mi       1.0Mi       600Mi       617Mi
Swap:             0B          0B          0B

从内存使用情况看，系统本身占用约 182 MB，缓存占用 600 MB，可用内存还有 617 MB。对于小内存 VPS 用户，建议手动配置 1-2 GB 的 Swap 分区作为缓冲，避免内存不足导致服务中断。

磁盘空间#

存储使用的是 20 GB 的 ext4 文件系统，全部挂载在根目录下。测试时系统已使用 4.7 GB，剩余可用空间为 15 GB。

root@r1089906:~# df -hT /
Filesystem     Type  Size  Used Avail Use% Mounted on
/dev/sda1      ext4   20G  4.7G   15G  25% /

虽然标称使用 NVMe 固态硬盘，但从设备名称 /dev/sda1 来看，这是传统的块设备命名方式（NVMe 通常显示为 nvme0n1）。不过实际读写性能才是关键，后续会通过磁盘 IO 测试来验证真实表现。对于建站用途，20 GB 空间可以运行博客或小型应用。

性能实测#

通过 sysbench 和 fio 等专业测试工具，对巴西圣保罗机房的硬件性能进行了全面测试。测试结果能够反映这台 VPS 在实际使用中的表现水平。

CPU 性能#

单核心 CPU 跑分测试显示每秒可完成 677.41 个事件，平均延迟 1.47 毫秒。这个成绩在入门级 VPS 中属于中等偏上水平。

root@r1089906:~# sysbench cpu --threads=1 run
CPU speed:
    events per second:   677.41
Latency (ms):
         min:                                    1.05
         avg:                                    1.47
         max:                                   11.11
         95th percentile:                        2.66

Intel Xeon E5-2699 v4 虽然是几年前的架构，但单核性能依然稳定。对于运行 WordPress、Typecho 等常规建站程序来说，这个 CPU 性能完全够用。不过如果需要编译大型项目或者运行复杂计算任务，单核心的限制会比较明显。

内存性能#

内存写入速度达到 3843.63 MiB/sec，每秒可完成近 400 万次操作。这个表现说明内存读写不会成为系统瓶颈。

root@r1089906:~# sysbench memory run
Total operations: 39367302 (3935879.55 per second)
38444.63 MiB transferred (3843.63 MiB/sec)

对于 1GB 内存的配置来说，内存速度本身不是问题，更需要关注的是合理分配和使用。建议运行内存占用较大的应用时，通过优化配置来降低内存消耗。

磁盘 IO 性能#

磁盘性能测试分为 4K 随机读写和顺序读写两个场景。4K 随机读写的 IOPS 达到 3364，对于 NVMe 固态硬盘来说偏低。

root@r1089906:~# fio --name=randrw --rw=randrw --bs=4k
  read: IOPS=3364, BW=13.1MiB/s
  write: IOPS=3363, BW=13.1MiB/s
  clat (usec): min=76, max=94639, avg=258.12

真正的 NVMe SSD 在 4K 随机读写场景下，IOPS 通常能达到几万甚至十几万。当前测试结果更接近 SATA SSD 的表现，虽然设备名称显示为 /dev/sda1，但从延迟和吞吐量来看，应该使用的是 SATA 接口的固态硬盘，而非真正的 NVMe 协议硬盘。

顺序读写测试表现相对更好，读取速度 481 MiB/s，写入速度 512 MiB/s。

root@r1089906:~# fio --name=seqrw --rw=readwrite --bs=1M
  read: IOPS=480, BW=481MiB/s
  write: IOPS=511, BW=512MiB/s

这个顺序读写速度已经能满足大部分建站需求，加载静态文件、数据库读写等操作都不会有明显卡顿。虽然宣传使用 NVMe 硬盘，但实际表现更像是性能不错的 SATA SSD。对于普通用户来说，这个磁盘性能已经够用，和高端 NVMe 还是有差距。

网络带宽#

通过 Speedtest 测试本地巴西网络，下载速度 390.97 Mbps，上传速度 416.08 Mbps，基本达到了标称的 300 Mbit 带宽上限。

root@r1089906:~# speedtest
Idle Latency:     2.08 ms
    Download:   390.97 Mbps
      Upload:   416.08 Mbps
 Packet Loss:     0.0%

本地测试延迟仅 2.08 毫秒，丢包率为零，说明巴西本地网络质量优秀。对于服务南美用户的网站或应用来说，这个带宽和延迟表现非常理想。需要注意的是，fair-share 带宽机制意味着高峰期可能会有一定的速度波动。

中国访问测试#

对于国内用户来说，巴西圣保罗机房的网络表现是最关心的问题。通过多个维度的测试，可以了解这个南美节点在访问国内网站和被国内用户访问时的真实状况。

回国延迟表现#

从巴西圣保罗 ping 国内百度服务器，延迟稳定在 329 毫秒左右，丢包率为零。

root@r1089906:~# ping -c 4 www.baidu.com
64 bytes from 103.235.46.102: icmp_seq=1 ttl=43 time=329 ms
64 bytes from 103.235.46.102: icmp_seq=2 ttl=43 time=329 ms
64 bytes from 103.235.46.102: icmp_seq=3 ttl=43 time=329 ms
64 bytes from 103.235.46.102: icmp_seq=4 ttl=43 time=329 ms
--- www.wshifen.com ping statistics ---
4 packets transmitted, 4 received, 0% packet loss
rtt min/avg/max/mdev = 328.720/328.740/328.764/0.020 ms

这个延迟数据符合地理位置的预期，巴西到中国的物理距离超过 17000 公里，光缆传输本身就需要较长时间。300 毫秒以上的延迟对于需要实时交互的应用（如游戏、视频会议）会有明显影响，但对于网站浏览、API 调用等场景还能接受。

国内各地延迟分布#

通过 itDog 平台测试国内多个节点到巴西圣保罗的延迟，整体平均延迟为 365 毫秒。

从三大运营商的表现来看，电信网络表现相对最好，联通部分节点延迟接近 500 毫秒。不同地区差异明显，沿海发达地区的延迟普遍低于内陆和西部地区。如果网站主要面向国内用户，这个延迟会影响页面加载速度和用户体验。

丢包率测试#

国内多节点测试的平均丢包率为 1.85%，这个数据在跨洲际线路中处于可接受范围。

少量丢包不会严重影响 HTTP/HTTPS 网页访问，TCP 协议会自动重传丢失的数据包。但对于 UDP 协议的应用（如语音通话、直播推流），1.85% 的丢包率可能会造成偶尔卡顿或画质下降的波动。

路由跟踪分析#

从中国访问巴西圣保罗的网络路径显示，数据包需要经过美国多个城市中转。

路由路径为：中国上海 → 美国圣何塞 → 旧金山 → 芝加哥 → 纽约 → 巴西圣保罗

这条路径说明中国到巴西没有直连光缆，需要绕道美国进行数据中转。每次跳转都会增加延迟，这也解释了为什么延迟会在 300 毫秒以上。这种路由结构对于国内用户访问巴西服务器来说比较常见，短期内难以改善。

实际下载速度#

使用国内 100Mbps 电信宽带通过 iperf3 测试从巴西服务器下载文件，平均速度约 50 Mbps。

速度从初期的几百 Kbps 逐渐爬升到稳定的 60-70 Mbps，整体平均 50 Mbps 左右。这个速度能够跑满国内 100M 宽带的一半带宽，表现比预期要好。测试过程中没有出现重传（Retr=0），说明连接稳定性不错。

IP 质量与流媒体解锁#

IP 质量检测#

通过专业的 IP 质量检测工具对巴西圣保罗节点进行全面扫描，结果显示这是一个典型的数据中心 IP。

IP 归属于 AS26383 ASNET，各大数据库均识别为机房类型。风险评分方面，IP2Location 和 AbuseIPDB 给出低风险评级，但 IPQS 评分达到 82 分显示为可疑 IP，Scamalytics 评为中等风险。

邮件服务方面，25 端口出站可用，Outlook、Yahoo 等主流邮局可正常通信，但也有部分邮局连接失败。IP 地址在 439 个黑名单数据库中检测，398 个显示正常，41 个已标记但未列入黑名单。整体来看，IP 质量中规中矩，适合一般建站使用。

流媒体解锁情况#

针对主流流媒体平台进行解锁测试，结果显示巴西本地服务可以正常访问。

从测试结果看，这个 IP 对巴西本地流媒体支持较好，但部分国际平台存在限制。如果需要稳定观看特定流媒体服务，建议提前确认该平台在巴西节点的解锁状况。

总结与建议#

JustHost 作为一家提供全球多机房服务的主机商，巴西圣保罗数据中心在南美地区具有明显的地理位置优势。

硬件配置方面，虽然宣传使用 NVMe 硬盘，但实测性能更接近 SATA SSD 水平。CPU 和内存表现中规中矩，符合入门级 VPS 的定位。网络带宽在本地测试中接近 400 Mbps，基本达到标称值。

对于国内用户来说，这个机房的主要短板在于回国延迟较高（300 毫秒以上），且需要经过美国中转，不适合面向国内用户建站。但如果业务需要巴西 IP、服务南美市场，或者用于学习测试，这个节点还是有一定价值的。
